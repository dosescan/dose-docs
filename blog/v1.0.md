## Why DOSE exists

Most security problems are easier to fix while the change is still small. DOSE focuses on that moment: right before a commit, right before a push, or right before a project is reviewed for release.

Instead of trying to replace dependency scanners, code review, or deeper security testing, DOSE works as a fast local guardrail. It looks for high-signal patterns that should rarely pass unnoticed: exposed credentials, risky shell execution, disabled verification, suspicious network operations, and other code that deserves a second look.

## Built for the commit loop

The default workflow is simple:

```bash
dose-scan --staged
```

This scans the files staged in Git, so the result matches what is actually about to be committed. Critical findings return a blocking exit code, while warning-only findings stay visible without stopping the workflow.

For wider checks, run:

```bash
dose-scan --all
```

Full scans recursively inspect readable text files while skipping common dependency, build, generated, and ignored directories.

## Install once, keep moving

macOS and Linux:

```bash
curl -sSL https://scan.softvenceomegaforce.cloud/install.sh | bash
```

Windows PowerShell:

```powershell
irm https://scan.softvenceomegaforce.cloud | iex
```

After installation, verify the command:

```bash
dose-scan --version
```

DOSE can also update itself from the stable release channel:

```bash
dose-scan --update
```

## Git hooks in one command

DOSE can wire itself into a repository with:

```bash
dose-scan --init
```

The generated pre-commit hook runs staged scanning. The generated pre-push hook runs a full audit. This gives teams a local feedback loop before code reaches the remote repository.

## Detection areas

DOSE 1.0 includes pattern groups for:

| Area | Examples |
| --- | --- |
| Dangerous execution | `eval`, process spawning, shell execution, dynamic imports |
| Secrets exposure | API keys, bearer tokens, private keys, database URLs |
| Data exfiltration | suspicious `curl`, `wget`, DNS tooling, reverse shell indicators |
| Unsafe practices | debug logs, weak crypto, disabled SSL verification, path traversal |
| Dependency risk | unpinned npm dependency ranges and deprecated package names |

The rule system is intentionally readable. Teams should be able to understand why something was flagged and decide whether to fix it, ignore it, or improve the rule.

## Designed to stay lightweight

DOSE is written as small shell modules instead of one large script. Scanner logic, Git integration, pattern definitions, CLI handling, file filtering, and installer behavior are separated so the project can grow without becoming difficult to maintain.

That design matters because security tools live inside daily workflows. If the tool is slow, noisy, or hard to reason about, developers route around it. DOSE 1.0 keeps the surface small: fast scans, focused output, and clear exit behavior.

## FAQ

### Is DOSE a replacement for full security review?

No. DOSE is a local guardrail for common dangerous patterns. It should be used alongside dependency scanning, code review, threat modeling, and production security controls.

### Does DOSE upload code?

No. DOSE is designed for local scanning. The documentation describes install and update flows, but scans run against files on the developer machine.

### Can teams customize what gets scanned?

Yes. Projects can use `.doseignore` to exclude files and directories from full scans, and `dose-ignore` comments for intentional line-level exceptions.

### Is this repository the scanner source code?

No. This public repository contains the documentation website. Public GitHub links should point to `dosescan/dose-docs`.
